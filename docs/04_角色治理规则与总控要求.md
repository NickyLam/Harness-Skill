# 角色治理规则与总控要求

## 1. 总控目标

这份规则不是让某个角色多做事，而是防止任何角色越界、跳步或无证据放行。

总控只管四件事：
- 任务怎么分级
- 角色怎么分工
- 哪些门禁不能跳
- 什么条件下可以收口

## 2. 五条硬规则

### 规则 1：先定入口，再进正式工作

每个任务都要先有任务入口单。

至少写清楚：
- 目标
- 范围内
- 范围外
- 验收标准
- 约束
- 未知问题
- 任务模式

### 规则 2：高风险内容先声明，再执行

下面这些内容不能静默进入执行：
- 接口约定变更
- 数据库结构、迁移或批量数据变更
- 权限、认证、敏感数据或安全配置变更
- 核心业务链路改动
- 批量、调度、消息链路改动
- 回退困难的改动
- 影响范围不清楚的改动

命中后必须先写高风险触发单。

### 规则 3：该独立的角色必须独立

下面这些组合不能混成同一个最终角色：
- 实施角色和最终验证角色
- 开发角色和安全审计角色
- 命中数据库风险时的开发角色和 DBA 审核角色
- 高风险任务中的实施角色和最终放行角色

### 规则 4：交接必须留下统一格式的书面结果

每次关键交接至少要有：
- 当前角色
- 当前阶段
- 输入工件
- 输出工件
- 验证证据
- 风险与未决项
- 交接状态
- 下一步

### 规则 5：没有证据，不能放行

下面这些材料缺任何一种，就不能把任务当成“已经完成”：
- 验证结果
- 关键工件
- 风险说明
- 回退说明
- 需要时的安全结论
- 需要时的数据库结论

## 3. 必须停下的情况

出现下面任一情况时，必须停下：
- 目标、范围、验收标准还不清楚
- 高风险动作还没先声明
- 必需工件缺失
- 必需证据缺失
- 测试没有通过
- 安全问题未处理
- 数据库问题未处理
- 影响范围说不清楚
- 回退方案说不清楚
- 必须独立的角色没有独立

停下后，必须补写：
- 卡点
- 原因
- 处理人
- 最小下一步

## 4. 高风险触发最小要求

高风险触发单至少要写：
- 改动内容
- 高风险原因
- 谁提出继续做
- 谁同意继续做
- 需要哪些额外角色
- 最低验证要求
- 回退准备

没有这张单，不进入正式执行。

## 5. 每阶段最低输出

| 阶段 | 最低输出 |
|---|---|
| 任务入口 | 任务入口单 |
| 需求澄清 | 需求说明 |
| 拆解与安排 | 任务拆解 |
| 方案决定 | 方案决策 |
| 实施与自检 | 实现交接 |
| 独立验证 | 测试结论 |
| 安全审核 | 安全审查 |
| 数据库审核 | 数据库审核 |
| 收口 | 项目收口 |

## 6. 收口条件

只有在下面条件都满足时，才能收口：
- 工件齐
- 证据齐
- 风险已写清楚
- 回退已写清楚
- 下一步已明确

否则只能写：
- 条件可交接
- 不可交接

不能写“完成”。
