# 角色治理规则与总控要求

## 1. 总控目标

这份规则不是让某个角色多做事，而是防止任何角色越界、跳步或无证据放行。

总控只管四件事：
- 任务怎么分级
- 角色怎么分工
- 哪些门禁不能跳
- 什么条件下可以收口

## 2. 六条硬规则

### 规则 0：先检查平台是否支持 sub-agent

- 平台支持时，默认按一角色一 sub-agent 执行
- 激活角色必须先主动尝试发起本角色 sub-agent，不允许直接跳过
- 如果因为模型可用性或平台瞬时错误失败，先重试，累计失败 5 次后才允许降级
- 平台不支持时，才允许由主控串行模拟角色
- 不能在支持 sub-agent 的平台上仍然把多角色混成一个执行单元

### 规则 1：先定入口、先清需求，再进技术方案和实现

每个任务都要先有任务入口单。

至少写清楚：
- 目标
- 主要场景
- 范围内
- 范围外
- 验收标准
- 约束
- 未知问题
- 任务模式

在需求内容、主要场景和验收标准未确认前：
- 不能先冻结技术栈
- 不能先冻结架构方案
- 不能先安排正式实现
- 需求澄清应优先逐个关键问题确认，不要一次抛出一串待确认问题
- 关键未知问题要继续和任务提出方沟通，不能直接当成已确认结论写入需求说明
- 不能把需求说明交接给下一角色
- 涉及 UI 变化时，不能在 UI 原型方向、参考案例或原图责任未确认前放行进入实现

### 规则 2：高风险内容先声明，再执行

下面这些内容不能静默进入执行：
- 接口约定变更
- 数据库结构、迁移或批量数据变更
- 权限、认证、敏感数据或安全配置变更
- 核心业务链路改动
- 批量、调度、消息链路改动
- 回退困难的改动
- 影响范围不清楚的改动

命中后必须先写高风险触发单。

补充要求：
- 架构方案必须给出多个可行方案的优劣比较
- 架构交付里必须包含架构体系图
- 架构方案形成后，必须由任务提出方明确确认最终方案；架构师只负责设计和推荐

### 规则 3：该独立的角色必须独立

下面这些组合不能混成同一个最终角色：
- 实施角色和最终验证角色
- 开发角色和安全审计角色
- 命中数据库风险时的开发角色和 DBA 审核角色
- 高风险任务中的实施角色和最终放行角色

补充要求：
- 项目经理必须要求发生变更的角色严格遵守 OpenSpec
- 除项目经理的 OpenSpec 门禁外，其他角色可灵活使用适合的 superpower 系列技能辅助执行

### 规则 4：交接必须留下统一格式的书面结果

每次关键交接至少要有：
- 当前角色
- 当前阶段
- 输入工件
- 输出工件
- 落盘路径
- 已完成内容
- 未完成内容
- 验证证据
- 风险与未决项
- 交接状态
- 下一步

如果交接内容由 sub-agent 辅助产出，当前责任角色仍然必须复核并署名负责。
每个角色阶段结束时都要交接，不允许只保留开发到测试这一段交接。

补充要求：
- 某角色检查不通过时，默认回流给上一个产出该工件的责任角色返工
- 项目经理只负责协调顺序和门禁，不替代上一个责任角色返工

### 规则 5：没有证据，不能放行

下面这些材料缺任何一种，就不能把任务当成“已经完成”：
- 验证结果
- 测试工程师测试结论
- 项目、服务或页面的启动或运行验证结果，或不适用说明
- 基本功能测试和集成测试结果，或不适用说明
- 关键工件
- 风险说明
- 回退说明
- 需要时的安全结论
- 需要时的数据库结论

## 3. 必须停下的情况

出现下面任一情况时，必须停下：
- 目标、范围、验收标准还不清楚
- 高风险动作还没先声明
- 必需工件缺失
- 必需证据缺失
- 关键未知问题还没与任务提出方沟通清楚
- 需求说明还没得到任务提出方确认却准备进入下一角色
- 涉及 UI 变化但 UI 原型方向、参考案例或原图责任未确认
- 发生了任何改动，但还没有测试工程师测试结论
- 发生了任何改动，但当前责任角色没有按 OpenSpec 组织和记录本阶段工作
- 测试没有通过
- 项目、服务或页面还没有通过启动或运行验证
- 平台支持 sub-agent，但激活角色没有先尝试发起 sub-agent 就直接降级
- 安全问题未处理
- 数据库问题未处理
- 影响范围说不清楚
- 回退方案说不清楚
- 必须独立的角色没有独立

停下后，必须补写：
- 卡点
- 原因
- 处理人
- 最小下一步

## 4. 高风险触发最小要求

高风险触发单至少要写：
- 改动内容
- 高风险原因
- 谁提出继续做
- 谁同意继续做
- 需要哪些额外角色
- 最低验证要求
- 回退准备

没有这张单，不进入正式执行。

## 5. 每阶段最低输出

| 阶段 | 最低输出 |
|---|---|
| 任务入口 | 任务入口单 |
| 需求澄清 | 需求说明 |
| 拆解与安排 | 任务拆解 |
| 方案设计 | 方案设计 |
| 实现准备 | 实现准备 |
| 实施与自检 | 实现交接 |
| 集成与联调 | 联调结论 |
| 独立验证 | 测试结论 |
| 安全审核 | 安全审查 |
| 数据库审核 | 数据库审核 |
| 收口 | 项目收口 |

补充要求：
- 每个阶段结束时，还要有一份角色交接记录
- 目标项目只保存业务工件和交接记录，不复制 Skill 包规则文档
- 方案设计阶段应补架构体系图

## 6. 收口条件

只有在下面条件都满足时，才能收口：
- 工件齐
- 证据齐
- 风险已写清楚
- 回退已写清楚
- 下一步已明确

否则只能写：
- 条件可交接
- 不可交接

不能写“完成”。
