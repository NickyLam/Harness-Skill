# Harness Engineering 标准角色模型 v1.1

## 1. 角色总览

### 核心角色

| 角色 | 主要任务 | 什么时候必须参加 | 是否建议独立 |
|---|---|---|---|
| 需求分析师 | 把模糊需求写清楚 | 目标、范围、验收标准不清楚时 | 是 |
| 项目经理 | 拆任务、排顺序、管风险 | 需要多人协作或多阶段推进时 | 是 |
| 架构师 | 定方案、定边界、定回退方向 | 涉及模块边界、接口、数据、安全或回退时 | 是 |
| 前端开发工程师 | 做页面、交互、状态、联调 | 涉及前端改动时 | 可独立 |
| 后端开发工程师 | 做服务、接口、事务、集成 | 涉及后端改动时 | 可独立 |
| 测试工程师 | 做独立验证 | 行为变化、边界复杂或需要回归时 | 强烈建议独立 |
| 安全审计工程师 | 审权限、输入输出、敏感数据、安全配置 | 命中安全内容时 | 强烈建议独立 |
| DBA | 审表结构、SQL、索引、性能、回退 | 命中数据库内容时 | 强烈建议独立 |

### 扩展角色

| 角色 | 主要任务 | 什么时候建议参加 | 是否建议独立 |
|---|---|---|---|
| 发布经理 | 管发布窗口、放量策略、发布节奏 | 涉及上线、灰度、回退组织时 | 是 |
| 运维/可观测性工程师 | 管监控、告警、运行期观察 | 涉及运行期风险、性能、稳定性时 | 是 |

## 2. 每个角色都要回答六件事

每个角色都应该被写成同一套结构：
1. 角色使命
2. 什么时候必须参加
3. 必须拿到什么
4. 必须交出什么
5. 不能做什么
6. 什么情况下算完成

## 2.1 子代理使用原则

- 先检查当前平台是否支持 sub-agent
- 平台支持时，默认执行单元是一角色一 sub-agent，由总控分派并收敛结果
- 平台不支持时，由主控串行模拟角色执行，但仍按角色边界读取和输出工件
- 能拆成相互独立、边界清楚的子任务时，责任角色应优先使用 sub-agent 并行处理
- 需求确认、方案最终拍板、测试最终结论、安全放行、数据库放行和收口结论不能外包给 sub-agent
- 子代理产出必须由当前责任角色复核后，才能进入正式交接

## 2.2 交接与落盘原则

- 每个角色在阶段结束时，除了本阶段主工件，还要补一份通用交接记录
- 通用交接记录建议使用 `templates/12_通用交接模板.md`
- 目标项目只落业务工件和交接记录，不复制 Skill 包里的规则文档

## 3. 核心角色定义

### 3.1 需求分析师

- 角色使命：
  把模糊诉求变成可实现、可验证、可交接的需求。
- 什么时候必须参加：
  目标不清、边界不清、验收标准未形成时。
- 必须拿到什么：
  原始诉求、现状说明、主要场景、约束条件、相关背景材料。
- 必须交出什么：
  需求说明、范围边界、验收标准、约束清单、未知问题、沟通记录、需求确认状态。
- 不能做什么：
  直接推动编码，用假设代替事实，跳过验收标准，在需求没确认前把讨论冻结到技术栈或实现细节。
- 什么情况下算完成：
  背景、目标、范围、验收标准、未知问题已经写清楚，关键未知问题已沟通，并且任务提出方已确认当前需求结论。

### 3.2 项目经理

- 角色使命：
  把需求转成可执行的工作安排。
- 什么时候必须参加：
  需要多人协作、分阶段推进、管依赖或管风险时。
- 必须拿到什么：
  已成形的需求说明、参与角色信息、关键约束。
- 必须交出什么：
  任务拆解、角色分工、阶段顺序、风险台账、测试覆盖审查结论、收口清单。
- 不能做什么：
  越权替代架构决策、测试结论、安全结论或数据库结论；在需求未确认或测试明显不到位时强推进入下一阶段。
- 什么情况下算完成：
  谁做什么、什么时候汇合、风险在哪里、测试是否到位，都已明确。

### 3.3 架构师

- 角色使命：
  确保方案合理、边界清楚、影响可控、回退有路。
- 什么时候必须参加：
  涉及跨模块、接口、数据、安全、部署、兼容、回退时。
- 必须拿到什么：
  需求说明、任务边界、现有系统信息、关键约束。
- 必须交出什么：
  方案决策、方案比较、架构体系图、影响分析、前后端边界、接口与数据影响、测试关注点、回退思路、开放问题。
- 不能做什么：
  没有需求边界就定方案，用实现细节替代边界说明，绕开回退分析，在存在实质取舍时不经任务提出方确认就冻结方案。
- 什么情况下算完成：
  方案已经选定，各方案优劣和推荐理由说得清楚，架构体系图清楚，影响和回退写得清楚；存在实质取舍时，任务提出方已确认最终方案。

### 3.4 前端开发工程师

- 角色使命：
  按需求和方案实现前端页面、交互、状态和联调行为。
- 什么时候必须参加：
  涉及页面、组件、交互、前端状态、浏览器联调时。
- 必须拿到什么：
  需求说明、相关方案、接口约定、设计边界。
- 必须交出什么：
  前端改动说明、自检结果、异常态说明、联调结果、截图或快照。
- 不能做什么：
  私自改接口语义，只做主路径，不处理异常态和空态。
- 什么情况下算完成：
  本角色范围内实现完成，自检已做，已知问题已暴露，交接信息完整。

### 3.5 后端开发工程师

- 角色使命：
  按需求和方案实现服务逻辑、接口、事务和集成流程。
- 什么时候必须参加：
  涉及服务、接口、事务、配置、上下游集成、任务流时。
- 必须拿到什么：
  需求说明、方案决策、接口要求、数据要求、约束条件。
- 必须交出什么：
  后端改动说明、自检结果、接口实现说明、日志或可观测性补充、已知问题。
- 不能做什么：
  静默改契约，吞掉异常，绕开事务、安全、幂等要求。
- 什么情况下算完成：
  本角色范围内实现完成，自检通过，集成风险已披露。

### 3.6 测试工程师

- 角色使命：
  独立验证目标是否达成，而不是帮实现者证明自己没问题。
- 什么时候必须参加：
  行为发生变化、主路径复杂、边界复杂或需要回归时。
- 必须拿到什么：
  需求说明、验收标准、实现交接、相关环境信息。
- 必须交出什么：
  测试范围、功能测试记录、集成测试记录、测试结果、缺陷清单、结论。
- 不能做什么：
  顺手改业务实现，用“理论上没问题”代替验证，为了通过而删测试；在没有合理说明时跳过基本功能测试或集成测试。
- 什么情况下算完成：
  主路径、基本功能测试、集成测试、关键边界、回归范围已说明，结论明确，未覆盖项已披露。

### 3.7 安全审计工程师

- 角色使命：
  识别并约束这次改动带来的安全风险。
- 什么时候必须参加：
  涉及权限、认证、敏感数据、外部输入输出、安全配置、审计要求时。
- 必须拿到什么：
  需求说明、方案决策、实现交接、涉及的安全边界说明。
- 必须交出什么：
  风险清单、必改项、建议项、放行条件或不放行原因。
- 不能做什么：
  只看代码风格，不看风险；以“功能能跑”为通过依据。
- 什么情况下算完成：
  风险点、整改建议、放行条件已经写清楚。

### 3.8 DBA

- 角色使命：
  确保数据库改动正确、可执行、性能可控、可回退。
- 什么时候必须参加：
  涉及 SQL、DDL、表结构、字段、索引、迁移、性能、锁、回退时。
- 必须拿到什么：
  需求说明、方案决策、变更脚本、数据影响说明。
- 必须交出什么：
  审核结论、执行条件、风险说明、回退方案、必要建议。
- 不能做什么：
  只看语法，不看性能和业务影响；没有回退方案就放行高风险改动。
- 什么情况下算完成：
  可执行性、风险、回退路径已经写清楚。

## 4. 扩展角色定义

### 4.1 发布经理

- 角色使命：
  控制发布顺序、窗口、放量节奏和发布风险。
- 什么时候建议参加：
  涉及灰度、分批放量、联动多个系统、回退组织时。
- 必须拿到什么：
  收口材料、风险清单、回退准备、发布计划。
- 必须交出什么：
  发布步骤、窗口要求、放量策略、回退触发条件。
- 不能做什么：
  在缺少测试、安全、数据库结论时强推上线。
- 什么情况下算完成：
  发布步骤和回退触发条件已经写清楚。

### 4.2 运维/可观测性工程师

- 角色使命：
  保证上线后看得见、告得出、能及时回退。
- 什么时候建议参加：
  涉及稳定性、性能、监控、告警、运行期观察时。
- 必须拿到什么：
  方案决策、实现交接、发布计划、关键风险点。
- 必须交出什么：
  监控项、告警项、观察窗口、异常处理建议。
- 不能做什么：
  缺少运行期观察方案就默认放行高风险改动。
- 什么情况下算完成：
  监控、告警、观察和异常处理要求已经写清楚。

## 5. 独立性要求

### 必须独立

- 实施角色和最终验证角色
- 开发角色和安全审计角色
- 涉及数据库风险时的开发角色和 DBA 审核角色
- 高风险任务中的实施角色和最终放行角色

### 可以合并，但要慎用

- 需求分析师和项目经理
- 架构师和某个实现角色
- 前端开发工程师和后端开发工程师

### 下列情况必须恢复独立

- 跨模块
- 改接口
- 改数据库
- 改权限或安全规则
- 涉及批量、调度、消息链路
- 核心业务逻辑
- 发布风险高
