# 平台无关约束原则

## 1. 定位

Harness Engineering 约束的不是“某个智能体怎么写代码”，而是“整个研发过程怎么被管住”。

它至少要回答这些问题：
- 任务怎么分级
- 每个阶段能做什么、不能做什么
- 高风险动作什么时候能开始
- 不同角色怎么交接
- 失败后怎么回退
- 在什么条件下才能说“完成”

这些规则应该尽量平台无关，避免把一套治理方法写死在某个具体工具里。

## 2. 七条总原则

### 原则 1：先写清楚，再进入正式实现

任何正式实现之前，都要先写清楚：
- 目标
- 范围内
- 范围外
- 验收标准
- 约束条件
- 未知问题

这些内容不清楚，就不能把任务当成“可直接实现”。

### 原则 2：高风险动作必须先声明，再执行

高风险动作不能默认自动进入执行。

必须先留下明确记录，说明：
- 改什么
- 为什么高风险
- 谁提出继续做
- 谁同意继续做
- 需要什么额外验证
- 回退准备是什么

### 原则 3：当前阶段只开放最小必要能力

不同阶段只能用完成当前阶段所需的最小能力。

建议分成四类能力：
1. 只读
2. 文档
3. 实现
4. 验证

建议映射：
- 分析阶段：只读
- 设计阶段：只读 + 文档
- 实现阶段：只读 + 文档 + 实现
- 验证阶段：只读 + 验证
- 收口阶段：只读 + 文档

### 原则 4：关键动作必须有前、中、后三次检查

关键动作不能只靠最后一次统一检查。

至少要有：
1. 开始前检查：能不能开始
2. 执行中检查：局部结果有没有偏掉
3. 结束后检查：工件、证据、风险、回退说明齐不齐

### 原则 5：失败反馈必须改变下一步动作

一旦出现下面这些失败，不能继续硬推：
- 构建失败
- 测试失败
- 契约校验失败
- SQL 校验失败
- 安全审查卡住
- 数据库审核卡住

失败后至少要做四件事：
- 解释失败点
- 判断原因
- 修改动作
- 重做相关验证

### 原则 6：没有证据，不能说完成

“完成”必须建立在证据上。

证据可以是：
- lint、build、test 结果
- 契约校验结果
- SQL 或迁移验证结果
- 页面截图或快照
- 安全审查结论
- 数据库审核结论
- 风险说明

没有证据，只能说：
- 已完成部分实现
- 待验证
- 已完成本阶段草稿

不能说“已经完成”。

### 原则 7：角色分工和独立验证不能被省略成口头约定

研发过程不能默认由一个智能体包办所有角色。

必须明确：
- 谁负责分析
- 谁负责安排
- 谁负责方案
- 谁负责实现
- 谁负责验证
- 谁负责安全
- 谁负责数据库
- 谁负责收口

角色之间必须通过标准工件交接，不能只靠会话记忆。

## 3. 任务分级原则

### 轻量模式

只有在下面条件全部满足时，才能判成轻量模式：
- 只影响单一局部
- 不改接口约定
- 不改数据库结构或批量数据
- 不改权限、认证、密钥或安全规则
- 回退简单
- 影响范围清楚

### 标准模式

标准模式用于普通研发任务。

适用情况：
- 行为会变化
- 需要多人配合
- 会改多个文件或一个模块
- 不满足轻量模式
- 也没有命中严格模式

### 严格模式

下面任一情况都要进入严格模式：
- 核心业务链路
- 接口约定变更
- 数据库结构、迁移或批量数据变更
- 权限、认证、敏感数据或安全配置变更
- 批量、调度、消息链路变更
- 回退困难
- 影响范围不清楚

如果拿不准，用更严格的模式。

## 4. 角色独立性原则

对于非小任务：
- 实施角色不得兼任最终验证角色
- 高风险任务中，实施角色不得兼任最终放行角色
- 涉及安全内容时，开发角色不得兼任安全审查角色
- 涉及数据库结构、迁移或 SQL 风险时，开发角色不得兼任 DBA 审核角色

## 5. 交接原则

每次关键交接至少要写清楚：
- 当前角色
- 当前阶段
- 输入工件
- 输出工件
- 已完成内容
- 未完成内容
- 验证证据
- 风险与未决项
- 交接状态
- 下一步

交接状态只有三种：
- 可交接
- 条件可交接
- 不可交接

## 6. 必须停下的情况

下面这些情况不能继续往前推：
- 目标、范围、验收标准不清楚
- 高风险动作还没先声明
- 缺少关键工件
- 缺少关键证据
- 失败结果还没处理
- 必须独立的角色没有独立
- 严格模式没有回退方案

如果停下了，就必须明确：
- 卡在哪
- 为什么卡住
- 最小下一步是什么
- 由谁来处理
