# 平台无关约束原则

## 1. 定位

Harness Engineering 不应只约束“一个智能体怎么开发”，还应约束：
- 软件工程过程有哪些阶段
- 每个阶段允许做什么
- 高风险动作如何进入
- 不同角色之间如何交接
- 失败反馈如何回流
- 在什么条件下才能宣称完成

这些约束应尽量采用平台无关表达，避免强绑定某一款 AI 编程平台。

---

## 2. 六条总原则

### 原则1：先有上下文和边界，再进入实现
任何开发任务在进入实现前，都应先明确：
- 任务目标
- 范围内/范围外
- 验收标准
- 约束条件
- 不确定点

没有清晰上下文时，不得直接进入正式实现。

### 原则2：高风险动作必须显式触发
高风险动作不得默认自动进入执行流程，必须由用户、工作流节点或审批动作显式触发。

高风险动作包括但不限于：
- 数据库结构变更
- SQL 批量修改
- 接口契约变更
- 配置中心变更
- 发布与回滚检查
- 权限与安全配置修改
- 核心业务链路改动

### 原则3：不同阶段只开放最小必要能力
AI 在不同工程阶段只能使用与当前阶段相匹配的最小必要能力，不得默认拥有全量执行能力。

建议抽象为四类能力：
1. 只读能力
2. 文档能力
3. 实现能力
4. 验证能力

建议阶段映射：
- 分析阶段：只读能力
- 设计阶段：只读 + 文档能力
- 实现阶段：只读 + 文档 + 实现能力
- 验证阶段：只读 + 验证能力
- 评审阶段：只读 + 文档能力

### 原则4：必须建立三段式控制
所有关键工程动作都必须纳入三段式控制：

1. 前置检查：确认进入条件满足
2. 执行后检查：立即收集局部验证反馈
3. 阶段结束检查：确认工件、证据、风险和评审信息完整

不得仅依赖一次性最终检查。

### 原则5：失败反馈必须改变下一步动作
一旦出现编译失败、测试失败、契约失败、SQL 校验失败、评审指出关键问题等反馈，
不得继续盲目前进，必须：
- 解释失败点
- 判断原因
- 采取修正动作
- 重做相关验证

### 原则6：没有证据，不得宣称完成
完成声明必须建立在证据之上，例如：
- lint / build / test 结果
- 契约校验
- SQL / 迁移验证
- 页面截图 / 快照
- 风险说明
- 评审摘要

没有证据，只能说“已完成部分实现”或“待验证”，不能说“已完成”。

---

## 3. 多角色原则

Harness Engineering 不应默认由单一智能体承担需求、计划、设计、实现、验证、审计和交付收口的全部职责。

必须引入角色分工，并通过以下维度约束角色：
- 职责范围
- 能力边界
- 输入工件
- 输出工件
- 完成条件
- 禁止行为

角色之间必须通过标准工件交接，而不是仅依赖隐含上下文或临时对话。

---

## 4. 角色独立性原则

非 trivial 任务中：
- 实现角色不得兼任最终验证角色
- 高风险任务中，实现角色不得兼任最终评审角色
- 涉及安全的任务中，开发角色不得兼任安全审计角色
- 涉及数据库结构或 SQL 风险的任务中，开发角色不得兼任 DBA 审核角色

---

## 5. 模式分级原则

### 轻量模式
适用：
- 小范围修复
- 文档/测试补充
- 不涉及接口/数据库/安全

### 标准模式
适用：
- 普通功能
- 一般缺陷修复
- 一般前后端协作任务

### 严格模式
适用：
- 核心业务
- 接口变更
- 数据库变更
- 安全敏感变更
- 高风险发布

模式越严格，参与角色越完整，验证和交付要求越高。
