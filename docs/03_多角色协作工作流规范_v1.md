# Harness Engineering 多角色协作工作流规范 v1.1

## 1. 目标

这份规范用来把一个任务从入口一直管到收口。

它重点约束：
- 任务怎么分级
- 哪些阶段不能跳
- 哪些角色必须独立
- 什么情况下能交接
- 什么情况下必须回退或升级

## 2. 总体流程

推荐总流程：

任务入口
→ 需求澄清
→ 任务拆解与安排
→ 方案决定
→ 实现准备
→ 实现与自检
→ 集成与联调
→ 独立验证
→ 安全与数据库审核
→ 收口与交付准备

不是每个任务都要把每个阶段拆得一样细，但门禁不能因为“图快”就消失。

## 3. 先做任务分级

### 轻量模式

只有在下面条件全部满足时，才能用轻量模式：
- 只影响单一局部
- 不改接口约定
- 不改数据库结构或批量数据
- 不改权限、认证、密钥或安全规则
- 回退简单
- 影响范围清楚

### 标准模式

默认模式。

适用：
- 普通功能
- 一般缺陷修复
- 常规多人协作任务

### 严格模式

命中下面任一条件就进入严格模式：
- 核心业务链路
- 接口约定变更
- 数据库结构、迁移或批量数据变更
- 权限、认证、敏感数据或安全配置变更
- 批量、调度、消息链路变更
- 回退困难
- 影响范围不清楚

如果拿不准，用更严格的模式。

## 4. 各阶段定义

### 阶段 0：任务入口与分级

- 责任角色：
  需求分析师或项目经理
- 最低输出：
  任务入口单
- 必须写清楚：
  目标、范围内、范围外、验收标准、约束、未知问题、任务模式
- 退出条件：
  任务模式已经明确，后续需要哪些角色已经初步明确

### 阶段 1：需求澄清

- 责任角色：
  需求分析师
- 协同角色：
  项目经理、架构师
- 最低输出：
  需求说明
- 退出条件：
  背景、目标、范围、验收标准、未知问题已写清楚

### 阶段 2：任务拆解与安排

- 责任角色：
  项目经理
- 协同角色：
  需求分析师、架构师、实施角色
- 最低输出：
  任务拆解
- 退出条件：
  谁做什么、顺序如何、依赖在哪里、汇合点在哪里，已经清楚

### 阶段 3：方案决定

- 责任角色：
  架构师
- 协同角色：
  项目经理、相关实施角色、需要时加入安全审计工程师和 DBA
- 最低输出：
  方案决策
- 下面这些情况不能跳过：
  - 跨模块
  - 改接口
  - 改数据库
  - 改安全规则
  - 影响回退
- 退出条件：
  方案已选定，原因、影响面和回退思路已写清楚

### 阶段 4：实现准备

- 责任角色：
  项目经理
- 协同角色：
  架构师、实施角色、测试工程师
- 最低输出：
  实现准备
- 退出条件：
  实施角色知道自己改什么，不改什么；测试工程师知道要验证什么

### 阶段 5：实现与自检

- 责任角色：
  前端开发工程师、后端开发工程师、需要时 DBA
- 协同角色：
  架构师、项目经理
- 最低输出：
  实现交接
- 退出条件：
  本角色范围内的实现已完成，自检已做，已知问题已披露

### 阶段 6：集成与联调

- 责任角色：
  相关实施角色
- 协同角色：
  架构师、项目经理、测试工程师、需要时 DBA
- 最低输出：
  联调结论
- 退出条件：
  主流程可跑通，阻塞问题已经解决或已经明确上升处理

### 阶段 7：独立验证

- 责任角色：
  测试工程师
- 协同角色：
  项目经理、相关实施角色
- 最低输出：
  测试结论
- 退出条件：
  主路径、关键边界、回归范围已说明，结论明确，未覆盖项已披露
- 如果结论为 FAIL 或存在阻塞缺陷：
  - 项目经理负责决定回流顺序和责任实施角色
  - 相关实施角色修复后必须重新提交实现交接
  - 测试工程师根据缺陷范围重新验证
- 如果结论为 PARTIAL：
  - 项目经理负责判断是否允许带条件进入下一阶段
  - 不能绕过未覆盖项和残留风险说明

### 阶段 8：安全与数据库审核

- 责任角色：
  安全审计工程师、DBA
- 协同角色：
  架构师、项目经理、相关实施角色
- 最低输出：
  安全审查、数据库审核
- 什么时候必须进入：
  - 命中安全内容
  - 命中数据库内容
  - 严格模式且相关风险存在
- 退出条件：
  风险、必改项、建议项、放行条件和回退条件已写清楚

### 阶段 9：收口与交付准备

- 责任角色：
  项目经理
- 协同角色：
  全部相关角色
- 最低输出：
  项目收口
- 退出条件：
  工件齐、证据齐、风险可见、回退可讲清楚、下一步明确

## 5. 三种模式如何裁剪流程

### 轻量模式

可以合并的阶段：
- 阶段 0 和阶段 1
- 阶段 1 和阶段 2
- 阶段 5 和阶段 6

不能省掉的要求：
- 仍然要有任务入口
- 行为变更时仍然要有验证结论
- 命中安全或数据库内容时仍然要有相应审核
- 入口角色由需求分析师或项目经理承担
- 行为无变化时，实施角色可以自证；行为有变化时，测试工程师负责验证

### 标准模式

可以并行，但不能把关键门禁抹掉：
- 拆解与安排可以和方案讨论交错推进
- 测试工程师可以提前准备验证范围
- 安全和 DBA 可以提前看风险点

不能省掉的要求：
- 需求、方案、实现、验证、收口都要留书面结果

### 严格模式

必须显式经过相关阶段。

额外要求：
- 执行前必须写高风险触发单
- 实施和验证必须独立
- 命中安全内容时安全审计不可省
- 命中数据库内容时 DBA 审核不可省
- 涉及上线时建议加入发布经理和运维/可观测性工程师

## 6. 角色激活条件

| 角色 | 激活条件 |
|---|---|
| 需求分析师 | 目标、范围、验收标准不清楚 |
| 项目经理 | 需要拆任务、排顺序、管风险、管交接 |
| 架构师 | 涉及边界、接口、数据、安全、回退 |
| 前端开发工程师 | 涉及页面、组件、交互、前端联调 |
| 后端开发工程师 | 涉及服务、接口、事务、配置、上下游集成 |
| 测试工程师 | 行为变化、边界复杂、需要回归 |
| 安全审计工程师 | 涉及权限、认证、敏感数据、外部输入输出、安全配置 |
| DBA | 涉及 SQL、表结构、索引、迁移、性能、锁、回退 |
| 发布经理 | 涉及上线窗口、灰度、分批放量、联动发布 |
| 运维/可观测性工程师 | 涉及监控、告警、稳定性、运行期观察 |

## 7. 交接协议

### 7.1 交接必须以工件为中心

不能只说“我做完了”。

每次交接至少要有：
- 当前角色
- 当前阶段
- 输入工件
- 输出工件
- 已完成内容
- 未完成内容
- 验证证据
- 风险与未决项
- 交接状态
- 下一步

### 7.2 三种交接状态

#### 可交接

同时满足下面条件：
- 本阶段必需输出齐全
- 没有阻塞性问题
- 证据足够支撑下一阶段继续
- 下一角色和下一步明确

#### 条件可交接

同时满足下面条件：
- 主结论已经成立
- 剩余事项只是小范围补充
- 不会改变主结论
- 不存在未处理的高风险问题
- 已明确谁补、补什么、补完后去哪一步

#### 不可交接

出现下面任一情况：
- 缺少必需工件
- 缺少关键证据
- 阻塞问题未解决
- 目标或范围不清楚
- 高风险问题未处理
- 必须独立的角色没有独立

## 8. 冲突、回退与升级

### 8.1 需求冲突

处理方法：
- 回到需求分析师
- 项目经理暂停后续依赖推进
- 需要时让架构师一起确认边界

### 8.2 计划冲突

处理方法：
- 项目经理重排顺序
- 收紧并行范围
- 需要时重开方案讨论

### 8.3 方案冲突

处理方法：
- 交给架构师裁决
- 影响需求目标时回到需求分析师
- 影响节奏时通知项目经理重排

### 8.4 实现冲突

处理方法：
- 先在实施角色范围内对齐
- 涉及接口、数据或安全边界时升级给架构师
- 影响计划时通知项目经理

### 8.5 验证冲突

处理方法：
- 不能私下口头覆盖验证结论
- 回到需求、方案、实现和证据逐项核对
- 安全风险不能由开发角色自行豁免
- 测试是否到位由项目经理检查是否覆盖验收标准、关键边界和必要回归
- 对测试范围是否足够存在争议时，回到需求分析师确认验收范围，回到架构师确认技术边界
- 项目经理可以要求补测，但不能替代测试工程师改写测试结论

### 8.6 数据库冲突

处理方法：
- 升级给架构师和项目经理
- 需要时要求重新做方案
- 没有回退方案时不能强推

### 8.7 高风险升级

命中下面任一条件时，从普通流升级到高风险流：
- 核心业务链路
- 接口约定变更
- 数据库结构、迁移或批量数据变更
- 权限、认证、敏感数据或安全配置变更
- 批量、调度、消息链路变更
- 回退困难
- 影响范围不清楚

升级后必须增加：
- 高风险触发单
- 独立角色参与
- 更强的验证说明
- 回退说明

## 9. 状态建议

建议统一使用下面这些状态：
- 任务入口中
- 需求澄清中
- 待拆解
- 待方案决定
- 待实施
- 实施中
- 联调中
- 待验证
- 验证中
- 待安全审核
- 待数据库审核
- 待收口
- 可交付
- 已阻塞
- 需回退
