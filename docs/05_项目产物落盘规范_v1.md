# Harness Engineering 项目产物落盘规范 v1

## 1. 规则与项目产物的边界

- `skills/`、`docs/`、`templates/` 属于 Skill 包自身，不属于目标项目的业务产物
- 目标项目不应把这套规则文档整体复制进工程目录
- 目标项目只落本次任务实际产生的业务工件、交接记录和收口材料

初始化目标项目时，应额外在项目根目录创建一份 `AGENTS.md`，并结合 OpenSpec 目录规范初始化项目骨架。

## 2. 初始化目标项目

初始化时至少创建：
- 项目根目录 `AGENTS.md`
- `openspec/` 目录
- `project-docs/` 产物目录
- `project-docs/02_planning/role-agent-registry.md`

`AGENTS.md` 建议基于 `templates/13_AGENTS模板.md` 生成，只保留项目级约束，不复制整套 Skill 包规则文档。

## 3. 推荐目录

建议采用“双轨目录”：
- `openspec/` 负责变更提案、设计、任务拆分和长期规格沉淀，是变更事实定义的唯一来源
- `project-docs/` 负责实现证据、测试结论、交接记录和收口材料，是执行证据的唯一来源

推荐先按 OpenSpec 初始化：

```text
openspec/
  config.yaml
  changes/
    archive/
  specs/
```

常用映射：

| OpenSpec 位置 | 说明 |
|---|---|
| `openspec/config.yaml` | OpenSpec 项目级配置 |
| `openspec/changes/<change>/proposal.md` | 本次变更的范围、目标、风险、为什么做 |
| `openspec/changes/<change>/design.md` | 本次变更的方案设计、边界、权衡 |
| `openspec/changes/<change>/tasks.md` | 本次变更的任务拆解和执行清单 |
| `openspec/specs/<capability>/spec.md` | 长期生效的能力规格、验收规则、约束 |
| `openspec/changes/archive/` | 已收口变更的归档位置 |

建议在目标项目内统一使用 `project-docs/` 作为产物根目录：

```text
project-docs/
  00_intake/
  01_requirements/
  02_planning/
  03_solution/
  04_implementation/
  05_validation/
  06_handoffs/
  07_closure/
```

## 4. 目录用途

| 目录 | 说明 |
|---|---|
| `project-docs/00_intake/` | 任务入口、分级、高风险触发单 |
| `project-docs/01_requirements/` | 需求说明、需求确认记录 |
| `project-docs/02_planning/` | 任务拆解、阶段安排、实施准备 |
| `project-docs/02_planning/role-agent-registry.md` | 角色常驻代理台账，记录 agent 标识、状态、替代关系和关闭条件 |
| `project-docs/02_planning/openspec-audit.md` | OpenSpec 审计记录，记录当前阶段应有工件、更新状态和一致性检查结果 |
| `project-docs/03_solution/` | 方案设计、边界说明、用户确认记录 |
| `project-docs/04_implementation/` | 实现交接、联调结论、实现证据 |
| `project-docs/05_validation/` | 测试结论、安全审查、数据库审核 |
| `project-docs/06_handoffs/` | 每个角色阶段结束后的通用交接记录 |
| `project-docs/07_closure/` | 项目收口、发布说明、运行观察说明 |

## 5. AGENTS.md 最低内容

- 需求确认门禁
- 架构确认门禁
- 角色执行方式
- 角色代理台账位置和维护规则
- OpenSpec 目录
- 项目产物目录
- 交接要求
- 验证要求

## 6. 最低要求

- 变更开始前，先在 `openspec/changes/<change>/` 下建立 proposal / design / tasks
- 稳定能力约束需要沉淀到 `openspec/specs/`
- 阶段 1 前至少建立 `proposal.md`，阶段 3 前至少完成 `design.md`，阶段 4 前至少完成 `tasks.md`
- 需求范围、方案、任务拆解或实施边界变化时，必须先更新对应 OpenSpec 工件，再更新 `project-docs/`
- 每个阶段的主工件必须落到对应目录
- 每个角色阶段结束时，除了主工件，还要补一份通用交接记录到 `project-docs/06_handoffs/`
- 平台支持 sub-agent 时，必须维护角色代理台账，并在首次创建、阶段复用、替代重建和最终关闭时更新
- 项目经理每次阶段放行前，必须更新 `project-docs/02_planning/openspec-audit.md`
- 文件名应带阶段、角色和日期，避免覆盖

## 7. 文件命名建议

- `YYYY-MM-DD-task-intake.md`
- `YYYY-MM-DD-requirements-handoff.md`
- `YYYY-MM-DD-solution-design.md`
- `YYYY-MM-DD-frontend-implementation.md`
- `YYYY-MM-DD-test-result.md`
- `YYYY-MM-DD-architect-handoff.md`
- `YYYY-MM-DD-project-closure.md`
