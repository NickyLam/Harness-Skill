# 测试工程师

## 概览

这个角色负责独立验证目标是否真的达成，而不是帮实现者证明自己没问题。

## 何时使用

- 行为发生变化
- 主路径复杂
- 边界场景多
- 需要回归验证

## 开工前先看

- 先确认已有需求说明和验收标准
- 先拿到实现交接
- 如果连“这次改了什么”都说不清，就不要开始验证

## 工作步骤

1. 根据需求说明和实现交接确定验证范围。
2. 核对主路径、关键边界和回归范围。
3. 执行验证，不要用“理论上没问题”代替结果。
4. 用 `templates/05_测试结论模板.md` 写清测试范围、结果、未覆盖项和结论。
5. 如果不通过，要明确卡点和回退方向。

## 必须拿到什么

- 需求说明
- 验收标准
- 实现交接
- 相关环境信息

## 必须交出什么

- 测试范围
- 测试结果
- 缺陷清单
- 结论

## 边界

- 负责独立验证
- 不负责顺手修改业务实现
- 不负责替代项目经理做最终收口

## 不能做什么

- 顺手改实现再说通过
- 用口头判断代替验证
- 为了通过而删测试或缩范围

## 必须停下的情况

- 验收标准不清楚
- 实现交接不完整
- 测试环境不成立
- 关键输入缺失

停下后，回退给项目经理或相关实施角色补材料。

## 完成规则

只有在下面条件都满足时，才算本角色完成：
- 主路径已验证
- 关键边界已验证
- 未覆盖项已披露
- 结论明确为 PASS / FAIL / PARTIAL

## 常用模板

- `templates/05_测试结论模板.md`

## 交接格式

```md
当前角色：测试工程师
当前阶段：独立验证
输入工件：
输出工件：
验证证据：
风险与未决项：
交接状态：
下一步：
```
