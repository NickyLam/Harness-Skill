# Harness Governance

## 概览

这是总控 Skill，不替代其他角色。
它只负责：判断任务级别、必需角色、阶段放行和最低证据。

## 何时使用

出现任一情况就使用：
- 一个任务会跨多个阶段或需要多角色协作
- 任务涉及接口、数据库、安全、发布或回退

## 核心规则

1. 目标、主要场景、范围、验收标准、约束、未知问题没写清前，不冻结技术栈、方案或实现分工。
2. 需求澄清时，逐个确认关键问题，不要一次抛出一串问题。
3. 涉及页面、交互或视觉变化时，先确认 UI 原型方向、参考案例或原图责任，再进入正式实现。
4. 默认不允许一个角色包办整条责任链。
5. 实现交接后，只要有改动，就必须进入测试工程师验证。
6. 发生变更时，项目经理必须要求当前责任角色严格遵守OpenSpec。
7. 其他角色可灵活使用superpower系列技能辅助执行，但不能替代本角色判断、确认和交接。
8. 进入执行前，先检查当前平台是否支持 sub-agent。
9. 支持 sub-agent 时，由总控在角色进入阶段、功能迭代启动、缺陷回流和重测回流时主动调起或恢复对应角色常驻代理；同一项目内同一角色默认只保留一个常驻代理。
10. 首次创建后的角色 sub-agent 应持续复用到项目验收、收口和交付完成；阶段推进、回流修复、重测和补证据优先继续使用原角色 sub-agent。
11. 角色代理的创建、复用、替换和关闭必须写入项目内角色代理台账，不能只留在会话记忆里。
12. sub-agent 因可用性或瞬时错误失败，5 次后才允许降级；常驻代理异常结束时，先恢复，再补建替代代理。
13. 平台不支持时，才允许由主控串行模拟各角色，但仍要保持角色边界和独立交接。
14. 可并行的独立子任务优先用 sub-agent；最终判断、确认和交接不能外包。
15. 高风险动作必须先声明，再执行。
16. 关键交接必须有工件，不能只靠对话记忆。
17. 初始化项目时，创建 `AGENTS.md`、`openspec/` 和 `project-docs/`。
18. `openspec/` 是变更范围、方案和任务拆解的事实来源；`project-docs/` 是执行证据和交接记录的事实来源，二者不能互相替代。
19. 任何进入下一阶段的角色都必须先核对本阶段要求的 OpenSpec 工件是否存在、已更新且与当前工作一致。
20. 必需的 OpenSpec 工件缺失、过期或与当前实现不一致时，交接状态只能是“不可交接”。
21. 规则文档和模板留在Skill包；项目里只落业务工件和交接记录。
22. 角色检查不通过时，默认回流给上一个产出该工件的责任角色返工。
23. 问题修复必须进入对应角色Skill，不能由总控或无关角色代修。
24. 验证失败后，下一步必须改变。
25. 没有新鲜证据，不能说完成。

## 第一步：检查执行平台

- 先判断当前平台是否支持 sub-agent / 子代理
- 支持时，由总控按角色拆分独立 sub-agent 执行，并建立角色代理台账
- 首次激活某个角色时，总控先查是否已有常驻代理；有则复用，没有再创建
- 模型可用性或平台瞬时错误时，重试最多 5 次后再降级
- 已创建的角色代理默认持续到项目验收、收口和交付完成后再关闭
- 不支持时，由主控串行调度，但仍按角色边界读取和输出工件
- 这一步没做完前，不进入项目初始化和角色分派

## 第二步：初始化目标项目

- 在项目根目录创建 `AGENTS.md`
- 在项目内创建 `openspec/`
- 在项目内创建 `project-docs/`
- 在 `project-docs/02_planning/` 创建角色代理台账
- 在 `project-docs/02_planning/` 创建 OpenSpec 审计记录
- `AGENTS.md` 建议基于 `templates/13_AGENTS模板.md`
- 初始化没完成前，不进入任务入口和角色分派

## 第三步：判断任务级别

### 轻量模式

必须同时满足：
- 只影响单一局部
- 不改接口约定
- 不改数据库结构或批量数据
- 不改权限、认证、密钥或安全规则
- 回退简单
- 影响范围清楚

### 标准模式

默认模式。适用于：
- 行为会变化
- 需要多人协作
- 会改多个文件或一个模块
- 不满足轻量模式且没有命中严格模式

### 严格模式

命中任一条即进入：
- 核心业务链路
- 接口约定变更
- 数据库结构、迁移或批量数据变更
- 权限、认证、敏感数据或安全配置变更
- 批量、调度、消息链路变更
- 回退困难
- 影响范围不清楚

拿不准时，用更严格的模式。

## 第四步：决定必须参加的角色

### 轻量模式

至少需要：
- 需求分析师或项目经理，二选一负责入口和分级
- 一个实施角色
- 一个测试工程师

说明：
- 文档、文案或纯说明改动时，入口角色可以和实施角色合并
- 命中安全或数据库内容时，再加入安全审计工程师或 DBA

### 标准模式

通常需要：
- 需求分析师
- 项目经理
- 需要时加入架构师
- 实施角色
- 独立测试工程师
- 命中安全或数据库内容时，加入安全审计工程师或 DBA

### 严格模式

通常需要：
- 需求分析师
- 项目经理
- 架构师
- 实施角色
- 独立测试工程师
- 命中安全内容时必须有安全审计工程师
- 命中数据库内容时必须有 DBA
- 涉及上线、发布、监控时，建议加入发布经理和运维/可观测性工程师

角色 Skill 路径：
- `skills/harness-requirements-analyst/SKILL.md`
- `skills/harness-project-manager/SKILL.md`
- `skills/harness-architect/SKILL.md`
- `skills/harness-frontend-engineer/SKILL.md`
- `skills/harness-backend-engineer/SKILL.md`
- `skills/harness-test-engineer/SKILL.md`
- `skills/harness-security-auditor/SKILL.md`
- `skills/harness-dba/SKILL.md`

## 第五步：补齐必需工件

至少补齐相关部分：
- 任务入口：`templates/00_任务入口模板.md`
- 需求说明：`templates/01_需求说明模板.md`
- 任务拆解：`templates/02_任务拆解模板.md`
- 方案设计：`templates/03_方案决策模板.md`
- 实现准备：`templates/10_实现准备模板.md`
- 实现交接：`templates/04_实现交接模板.md`
- 联调结论：`templates/11_联调结论模板.md`
- 测试结论：`templates/05_测试结论模板.md`
- 安全审查：`templates/06_安全审查模板.md`
- 数据库审核：`templates/07_数据库审核模板.md`
- 项目收口：`templates/08_项目收口模板.md`
- 通用交接：`templates/12_通用交接模板.md`
- 项目初始化：`templates/13_AGENTS模板.md`
- 角色代理台账：`templates/14_角色代理台账模板.md`
- OpenSpec 审计：`templates/15_OpenSpec审计模板.md`

严格模式或高风险任务还要补：
- 高风险触发单：`templates/09_高风险触发模板.md`

目标项目应保留 `openspec/` 和 `project-docs/`；建议按 `docs/05_项目产物落盘规范_v1.md` 落盘。
OpenSpec 最低要求：
- 阶段 0 后至少有 `proposal.md`
- 阶段 3 后至少有并更新 `design.md`
- 阶段 4 后至少有并更新 `tasks.md`
- 需求范围、方案、任务拆解或实施边界变化时，先更新对应 OpenSpec 工件，再继续交接

## 第六步：检查阶段门禁

阶段最低要求：
- 阶段 0 任务入口：目标、主要场景、范围内、范围外、验收标准、约束、未知问题、任务模式写清
- 阶段 1 需求澄清：背景、目标、范围、验收标准、未知问题写清；关键未知问题已逐个沟通；任务提出方已确认需求结论；未确认前不得交接；且 `proposal.md` 已建立并与当前需求结论一致
- 阶段 1 需求澄清：涉及 UI 变化时，UI 原型方向、参考案例或原图责任已确认
- 阶段 2 拆解安排：只在需求确认后进行，角色分工、依赖、风险、汇合点写清；且 OpenSpec 变更目录和适用工件已明确
- 阶段 3 方案设计：备选方案优劣、推荐理由、架构体系图、前后端边界、接口契约、测试关注点、回退思路写清；任务提出方确认后才能交接；且 `design.md` 已更新到当前确认方案
- 阶段 4 实现准备：改什么、不改什么、联调前提、测试重点、前置确认状态、OpenSpec 执行要求写清；涉及 UI 变化时原型来源和确认状态写清；且 `tasks.md` 已更新到当前实现范围
- 阶段 5 实现与自检：实际改动、自检结果、启动或运行自检、已知问题、给测试的验证入口和下游核对点写清；涉及 UI 变化时效果确认结果写清；如实现边界有变化，必须先回写对应 OpenSpec 工件
- 阶段 6 集成联调：联调范围、结果、阻塞问题、回流对象写清
- 阶段 7 独立验证：只要实现交接存在任何改动，就必须进入本阶段；先输出测试案例再执行；功能测试必做，集成测试和非功能测试按需求内容判断；启动或运行检查、未覆盖项、结论写清；测试范围与结论必须与当前 OpenSpec 工件一致
- 阶段 8 安全与数据库审核：风险、必改项、建议项、放行条件写清
- 阶段 9 收口：工件、证据、风险、回退、下一步写清；并完成 OpenSpec 闭环审计
- 支持 sub-agent 时，阶段责任角色必须留下尝试、重试和是否降级的记录
- 支持 sub-agent 时，阶段责任角色必须先查角色代理台账；已有常驻代理则优先复用
- 角色代理台账至少记录：角色名、agent 标识、创建阶段、最近状态、最近工件、替代关系、关闭条件、关闭时间
- 回流修复、重测和补材料时，优先恢复并继续使用原角色常驻代理；无法恢复时才补建替代代理，并在台账写清原因
- 功能迭代进入实施前，总控必须先调起对应实施角色常驻代理
- 缺陷回流和重测回流开始前，总控必须先恢复或调起对应责任角色常驻代理
- 项目经理每次放行前都要补一份 OpenSpec 审计记录

每个阶段结束时：
- 当前责任角色都要补一份通用交接记录
- 交接记录写入目标项目的 `project-docs/06_handoffs/`

FAIL 或阻塞缺陷时：
- 默认回流给上一个产出该工件的责任角色返工
- 问题修复必须重新激活对应角色Skill；前端问题进前端，后端问题进后端
- 项目经理决定回流顺序、责任边界和重验入口
- 返工角色修复后重新提交本阶段主工件和通用交接
- 测试工程师或相关审核角色按问题范围重验

PARTIAL 时：
- 项目经理决定能否带条件进入下一阶段
- 不能绕过未覆盖项和残留风险说明

## 第七步：高风险动作必须先触发

执行前必须写高风险触发单，至少说明：
- 改什么
- 为什么高风险
- 谁提出继续做
- 谁同意继续做
- 需要哪些额外角色
- 最低验证要求
- 回退思路

没有这张单，不进入执行。

## 第八步：交接状态

只能使用三种状态：
- 可交接：本阶段输出齐全、无阻塞、证据足够、下一角色和下一步清楚
- 条件可交接：主结论成立、剩余只是小范围补充、无未处理高风险、补充责任清楚
- 不可交接：目标不清、缺工件、缺证据、有阻塞、独立性不满足、高风险未处理，或 OpenSpec 工件缺失 / 过期 / 不一致

## 第九步：必须停下的情况

出现任一情况就停下并回退或升级：
- 需求边界冲突
- 设计和目标冲突
- 涉及 UI 变化但 UI 原型方向、参考案例或原图责任还没确认
- 关键未知问题还没跟任务提出方沟通清楚
- 需求说明还没有得到任务提出方确认，但已经准备进入下一角色
- 架构方案还没有得到任务提出方明确确认，但已经准备进入实现准备或实现
- 项目还没有创建 `AGENTS.md`
- 当前阶段要求的 `proposal.md` / `design.md` / `tasks.md` 还没有建立
- OpenSpec 工件没有跟随本轮需求、方案、任务拆解或实现边界变化及时更新
- 当前交接只更新了 `project-docs/` 证据，没有回写 `openspec/` 事实定义
- 当前责任角色发生变更，但还没有按 OpenSpec 组织和记录本阶段工作
- 问题已经判定需要前端或后端修复，但还没进入对应角色Skill
- 实现交接已完成且存在改动，但测试工程师还没有明确输入或还没开始验证
- 测试没有通过
- 项目、服务或页面还没通过启动或运行验证
- 安全问题未处理
- 数据库风险未处理
- 严格模式没有回退方案
- 实施角色和必须独立的审核角色是同一个
- 影响范围说不清楚
- 平台支持 sub-agent 但还没有按角色拆分执行
- 平台支持 sub-agent，但激活角色没有先尝试就直接降级为主控串行执行
- 平台支持 sub-agent，但当前阶段没有先检查并复用该角色已存在的常驻代理
- 平台支持 sub-agent，但功能迭代、缺陷修复或重测回流前，总控没有先调起或恢复对应角色常驻代理
- 项目已经进入收口前阶段，但角色代理台账仍缺失或无法说明代理状态

## 第十步：完成声明规则

只有拿得出新鲜证据时，才能说完成。

阶段结束至少输出：

```md
当前角色：
当前阶段：
输入工件：
输出工件：
落盘路径：
角色代理：
已完成内容：
未完成内容：
验证证据：
风险与未决项：
交接状态：
下一步：
```

需要更完整说明时，再配合 `docs/` 和 `templates/`。
