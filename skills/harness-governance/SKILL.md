# Harness Governance

## 概览

这个 Skill 是总控层，不直接替代需求、设计、开发、测试、安全或数据库角色。

它的任务只有四件事：
- 判断这项工作现在属于什么级别
- 决定哪些角色必须参加
- 决定什么时候能继续、什么时候必须停下
- 决定每一步至少要留下什么证据

## 何时使用

遇到下面任一情况，就应该使用这个 Skill：
- 一个任务会跨过多个阶段
- 不止一个角色需要协作
- 需要判断是否能从当前阶段进入下一阶段
- 任务可能影响接口、数据库、安全、发布或回退
- 需要避免“开发自己说自己完成了”这种闭环

## 核心规则

1. 没有写清楚目标、范围、验收标准、约束和未知问题之前，不进入正式实现。
2. 默认不允许一个角色包办整条责任链。
3. 高风险动作必须先声明，再执行。
4. 每次关键交接都要留下书面工件，不能只靠对话记忆。
5. 验证失败后，下一步必须改变，不能继续硬推。
6. 没有新鲜证据，不能说“已经完成”。
7. 角色职责不清时，先停下，先分清，再继续。

## 第一步：判断任务级别

### 轻量模式

只有在下面条件全部满足时，才能用轻量模式：
- 只影响单一局部
- 不改接口约定
- 不改数据库结构或批量数据
- 不改权限、认证、密钥或安全规则
- 回退简单
- 影响范围说得清楚

常见例子：
- 文档修改
- 文案调整
- 测试补充
- 局部样式修正
- 影响很小、边界清楚的缺陷修复

### 标准模式

标准模式是默认模式。

适用情况：
- 行为会变化，但风险还可控
- 需要多人协作
- 会改多个文件或一个模块
- 不满足轻量模式
- 也没有命中严格模式

### 严格模式

只要命中任意一条，就必须进入严格模式：
- 核心业务链路
- 接口约定变更
- 数据库结构、迁移或批量数据变更
- 权限、认证、敏感数据或安全配置变更
- 批量、调度、消息链路变更
- 回退困难
- 影响范围不清楚

如果拿不准，用更严格的模式。

## 第二步：决定必须参加的角色

### 轻量模式

至少需要：
- 任务入口负责人
- 实施角色
- 验证角色

说明：
- 如果只是文档、文案或纯说明性改动，验证角色可以和实施角色合并
- 如果行为发生变化，验证角色最好独立
- 只有碰到安全或数据库内容，才额外拉入安全或数据库角色

### 标准模式

通常需要：
- 需求分析师
- 项目经理
- 需要时加入架构师
- 实施角色
- 独立测试工程师
- 命中安全或数据库内容时，加入安全审计工程师或 DBA

### 严格模式

通常需要：
- 需求分析师
- 项目经理
- 架构师
- 实施角色
- 独立测试工程师
- 命中安全内容时，必须有安全审计工程师
- 命中数据库内容时，必须有 DBA
- 涉及上线、发布、监控时，建议加入发布经理和运维/可观测性工程师

如果已经决定某个角色必须参加，就继续加载对应的角色 Skill：
- 需求分析师：`skills/requirements-analyst/SKILL.md`
- 项目经理：`skills/project-manager/SKILL.md`
- 架构师：`skills/architect/SKILL.md`
- 前端开发工程师：`skills/frontend-engineer/SKILL.md`
- 后端开发工程师：`skills/backend-engineer/SKILL.md`
- 测试工程师：`skills/test-engineer/SKILL.md`
- 安全审计工程师：`skills/security-auditor/SKILL.md`
- DBA：`skills/dba/SKILL.md`
- 发布经理：`skills/release-manager/SKILL.md`
- 运维/可观测性工程师：`skills/observability-engineer/SKILL.md`

## 第四步：检查阶段门禁

### 阶段 0：任务入口

必须写清楚：
- 目标
- 范围内
- 范围外
- 验收标准
- 约束
- 未知问题
- 任务模式

缺一项，就不要进入后续正式工作。

### 阶段 1：需求澄清

通过条件：
- 背景明确
- 目标明确
- 范围边界明确
- 验收标准明确
- 未知问题已列出

### 阶段 2：任务拆解与安排

通过条件：
- 谁做什么清楚
- 依赖关系清楚
- 风险清单已初始化
- 汇合点清楚

### 阶段 3：方案决定

在下面情况不能跳过：
- 跨模块
- 改接口
- 改数据库
- 改安全规则
- 影响回退

通过条件：
- 选择了哪种方案
- 为什么这么选
- 影响面是什么
- 回退思路是什么

### 阶段 4：实现与自检

实施角色必须交出：
- 实际改动说明
- 自检结果
- 已知问题
- 需要下游继续核对的点

### 阶段 5：独立验证

验证角色至少要说明：
- 验了什么
- 怎么验的
- 结果如何
- 哪些没覆盖
- 结论是通过、部分通过还是不通过

### 阶段 6：安全与数据库审核

命中相关内容时不能跳过。

通过条件：
- 风险清楚
- 必改项和建议项分开写清楚
- 放行条件明确

### 阶段 7：收口

只有在下面条件都满足时，才能说任务可以收口：
- 必要工件齐了
- 验证证据齐了
- 残留风险写清楚了
- 回退方案写清楚了
- 下一步是交付、继续修正还是暂停，已经明确

## 第五步：高风险动作必须先触发

高风险动作在执行前，必须先写一张高风险触发单，至少说明：
- 这次要改什么
- 为什么算高风险
- 谁提出继续做
- 谁同意继续做
- 需要哪些额外角色参加
- 最低验证要求
- 回退思路

没有这张单，不进入执行。

## 第六步：只能使用三种交接状态

### 可交接

只有在下面条件都满足时，才能写“可交接”：
- 本阶段必需输出已经齐全
- 没有阻塞性问题
- 已有足够证据支持进入下一阶段
- 下一角色和下一步动作清楚

### 条件可交接

只有在下面条件都满足时，才能写“条件可交接”：
- 主方向已经明确
- 剩下的是小范围补充，不会改变主结论
- 不存在未处理的高风险问题
- 已明确谁来补、补什么、补完后去哪一步

### 不可交接

出现下面任一情况，就必须写“不可交接”：
- 目标或范围还不清楚
- 缺少必需工件
- 缺少关键证据
- 阻塞问题未解决
- 该独立的角色没有独立
- 高风险内容还没被明确处理

## 第七步：必须停下的情况

出现下面任一情况，必须停止推进并回退或升级：
- 需求边界冲突
- 设计和目标打架
- 测试没有通过
- 安全问题未处理
- 数据库风险未处理
- 严格模式没有回退方案
- 实施角色和必须独立的审核角色是同一个
- 影响范围说不清楚

## 第八步：完成声明规则

只有拿得出新鲜证据时，才能说“已经完成”。

每次阶段结束，至少输出下面这些内容：

```md
当前角色：
任务模式：
当前阶段：
输入工件：
输出工件：
验证证据：
风险与未决项：
交接状态：
下一步：
```
