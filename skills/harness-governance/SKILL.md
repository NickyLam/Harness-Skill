# Harness Governance 总控 Skill

## 一、目标

本 Skill 不直接承担具体开发实现，而是作为 AI 软件工程的总控治理规则，
约束不同角色的职责、能力、交接与门禁，确保工程过程可读、可控、可验证、可回流。

## 二、标准角色

标准角色至少包括：
1. 需求分析师
2. 项目经理
3. 架构师
4. 前端开发工程师
5. 后端开发工程师
6. 测试工程师
7. 安全审计工程师
8. DBA

## 三、角色治理原则

### 原则1：不得默认由单一智能体承担全部职责
AI 软件工程不得默认由单一智能体同时承担需求分析、项目统筹、架构设计、实现开发、测试验证、安全审计和数据库审核全部职责。

### 原则2：每个角色必须有明确契约
每个角色必须显式定义：
- 角色使命
- 职责范围
- 输入工件
- 输出工件
- 能力边界
- 完成条件
- 禁止行为

### 原则3：角色之间必须通过标准工件交接
关键交接必须依赖标准工件，不得仅依赖隐含上下文、临时会话记忆或非结构化交流完成。

### 原则4：每个角色只能使用最小必要上下文
角色只应接收完成本角色工作所需的最小上下文，不得默认共享全量上下文。

### 原则5：角色不得越权闭环处理超出自身职责的问题
当某角色发现问题超出自身职责边界时，必须升级、回退或交接给相应角色处理。

## 四、阶段与责任链

标准责任链如下：
需求分析师
→ 项目经理
→ 架构师
→ 前端开发工程师 / 后端开发工程师 / DBA
→ 测试工程师
→ 安全审计工程师
→ 项目经理收口

可并行，但不得跳过关键门禁。

## 五、独立性要求

### 1. 实现与验证独立
非 trivial 任务中，开发角色不得兼任最终测试角色。

### 2. 实现与安全独立
涉及鉴权、权限、敏感数据、安全配置的任务中，开发角色不得兼任最终安全审计角色。

### 3. 实现与数据库审核独立
涉及 SQL、表结构、索引、迁移、回滚的任务中，开发角色不得兼任 DBA 审核角色。

### 4. 高风险任务必须多角色参与
高风险任务必须至少经过架构师、测试工程师、安全审计工程师、DBA 中相关角色的独立参与后，才能进入交付准备阶段。

## 六、角色完成定义

### 需求分析师完成条件
- 需求背景明确
- 范围边界明确
- 验收标准明确
- 约束清单明确
- 不确定点已披露

### 项目经理完成条件
- 任务拆解完成
- 依赖关系明确
- 参与角色明确
- 风险台账存在
- 阶段切换条件明确

### 架构师完成条件
- 技术方案明确
- 取舍说明明确
- 影响分析存在
- 兼容性说明存在
- 回滚思路存在

### 开发角色完成条件
- 功能按要求实现
- 自测完成
- 必要文档同步
- 未引入已知重大失败
- 未越界修改

### 测试工程师完成条件
- 主路径验证完成
- 关键边界验证完成
- 回归检查完成
- 验证结论明确
- 未覆盖项已说明

### 安全审计工程师完成条件
- 核心安全点已审
- 风险清单存在
- 整改建议存在
- 放行建议明确

### DBA 完成条件
- 数据影响清楚
- SQL 审核完成
- 性能/锁风险说明存在
- 回滚方案存在
- 执行条件明确

## 七、高风险升级规则

以下情况视为高风险：
- 涉及核心业务流程
- 涉及接口契约变更
- 涉及数据库结构变更
- 涉及权限或安全控制
- 涉及批量/调度/消息链路
- 涉及高影响面回滚困难变更

高风险任务必须：
- 进入显式触发流程
- 增加独立角色参与
- 增加验证与评审强度
- 输出残留风险说明
- 输出回滚说明

## 八、禁止行为

严禁以下行为：
1. 跳过需求边界直接开发
2. 无设计说明推动跨边界变更
3. 开发角色自证完成并直接放行
4. 安全和数据库风险未审即推进交付
5. 仅依赖会话记忆完成关键交接
6. 发现职责越界问题仍继续闭门处理
7. 没有证据就宣称任务完成

## 九、最终输出要求

每个阶段结束时，必须至少输出：
- 当前角色名称
- 本阶段输入工件
- 本阶段输出工件
- 完成情况
- 风险与未决项
- 是否可交接给下一角色

若不可交接，必须明确阻塞原因和最小下一步动作。
